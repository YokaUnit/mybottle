## 把握の全体像（どれくらい分かっているか）

| 領域 | 把握度 | 根拠 |
|------|--------|------|
| サービスコンセプト・戦略 | **高** | `docs/service-definition/mybottle-service-definition.md` が明確 |
| ユーザー向けコアフロー | **高** | 購入・消費・保有画面をコードで追跡済み |
| 店舗・運営側フロー | **中〜高** | staff 価格申請・admin 承認は実装＋仕様書あり。店員向け専用 UI は意図的に無し |
| データモデル | **中〜高** | Supabase テーブルはコードから逆引き可能。migration はリポジトリ外の可能性 |
| ビジネス数値・KPI | **低** | 売上目標・導入店舗数・検証結果の記録は未着手 |
| 本番運用・法務 | **低** | 利用規約ページはあるが、決済・個人情報の本番運用詳細は未確認 |
| UI/UX の意図 | **中〜高** | 会話でログイン・ホーム・詳細画面のデザイン意図を直接反映済み |

**一言で言うと:** 「何を作ろうとしているか」「なぜそう設計するか」はよく分かっている。一方で「市場で既に効いているか」「収益モデルの確定版」はリポジトリだけでは分からない。

## サービスとは何か

**mybottle**（YokaUnit 製）は、バー・ラウンジ・スナック等の **ボトルキープ** をスマホで管理する **来店促進サービス**。

### 一言定義（公式）

> **「店のやり方を変えずに来店とリピートを増やす仕組み」**

### 本質

- **DX しすぎない** — 伝票・店頭決済・店員オペはそのまま
- **サイトの役割** — 購入証明・保有/消費の状態管理・来店前の意思決定
- **店舗の役割** — 代金受取・伝票記入・提供

初期方針は **アプリ内クレジット決済なし（店頭決済）**。決済 DX ではなく **運用 DX** に集中する。

---

## 戦略（なぜこの形か）

### 最大の差別化: 「変えなくていい DX」

多くの店舗 SaaS が失敗するのは、現場フローを置き換えようとするから。mybottle は逆を取る。

| 当事者 | 従来 | mybottle 後 |
|--------|------|-------------|
| 店員 | 伝票に記入 | **そのまま**（画面を見て確認するだけ） |
| 店舗 | 現金/カード/QR で会計 | **そのまま** |
| ユーザー | 来店してボトルを選ぶ | **事前にアプリで確保・割引確認** → 来店時は画面提示 |

### ユーザーが使う理由（仮説・公式整理）

1. **得** — 同じボトルが安く買える（割引がフック）
2. **所有感** — 「自分のボトル」がスマホにある
3. **思考ゼロ** — 「これ見せれば OK」

ターゲットは **学生・若者** を強く意識（割引・ポップな UI・モバイルファースト）。

### 店舗が入る理由

1. **来店の事前確定** — 店選びの段階で勝てる
2. **常連化・リピート**
3. **導入コスト極小** — 教育・システム操作ほぼ不要

### 営業トーク（そのまま使えると定義されている）

- 「今のやり方そのままでいいです」
- 「伝票も今まで通り書いてください」
- 「お客さんは安く買えるので来店が増えます」
- 「操作は一切いらないです」
- 「支払いはこれまでの方法のままで大丈夫です」

### 地域・ポジション

コード・店舗データから **湘南エリア（茅ヶ崎・藤沢・平塚等）** 起点の展開が読み取れる。ドメイン想定は `mybottle.app`。

## コアユーザーフロー

### 購入（店頭決済前提）

1. 店舗を選ぶ（`/products/step-1`）
2. ボトル（商品）を選ぶ（step-2）
3. セット数を選ぶ（step-3）
4. 店頭で支払い → **PIN 等でアプリ上確定**（step-4）
5. `user_stock` に残量が加算される

※ 実決済は Stripe 等ではなく、**ユーザー自己申告 + 店員の目視確認** が前提。

### 消費（来店時の「使う」）

1. 店舗・ボトルを選ぶ（`/consume/step-1`）
2. 杯数を選ぶ（step-2）
3. **店員提示用画面**（トークン表示）→ 確定（step-3）
4. `remaining_units` 減算、`stock_events` に記録

ボトムナビ中央の **「使う」** がこのフローの入口。

### 保有・管理

- ホーム / `/bottles` — 一覧（店舗別横スクロール）
- `/bottle/[storeId]/[productId]` — 詳細（残量・有効期限・履歴）
- `/history` — 利用履歴
- `/add-bottle` — QR 導線（**現状 UI モック**）

### ログイン

- **Google OAuth**（Supabase）のみ本番実装
- `/login` — `start_page.png` 背景のランディング型ログイン
- 未ログインは `(main)` 配下をガード

---

## 店舗・運営側

### 公開

- `/staff` — 加盟店募集 LP（ログイン画面下部からも導線）

### staff ロール

- `/dashboard` — 運用ダッシュボード（指標はログベースの疑似値含む）
- `/dashboard/pricing` — **担当店舗の価格変更申請**（理由必須 → admin 承認）

### admin ロール

- `/admin` — ユーザー権限、staff↔店舗紐付け、価格申請承認/却下、店舗・商品・特典ニュース CRUD

### 価格の考え方

- 商品マスタ `products.price_jpy`（グローバル）と、店舗別 `store_products.current_price_jpy` が共存
- **承認フロー後は店舗別価格が正** という設計
- ⚠️ **購入フローがまだグローバル価格を参照している可能性** — 仕様と実装のギャップ

### 意図的に無いもの

- 店員専用の「購入確認タブレット」
- 店舗 POS 連携

→ 現場は **伝票 + ユーザーのスマホ画面** で回す。

## データ・ドメインモデル（把握している範囲）

| 概念 | 説明 |
|------|------|
| `stores` | 加盟店 |
| `products` | 飲料カテゴリ（`virtual` / `physical`、セット杯数 `bundle_size`） |
| `user_stock` | ユーザー×店舗×商品の **残量** |
| `stock_events` | purchase / consume / gift / remove 等のログ |
| `profiles.role` | `user` / `staff` / `admin` |

**有効期限:** クライアント側で `updatedAt + 3ヶ月` として表示（ビジネスルールのハードコード）。

**画像:** `public/bottles/` のスラッグ PNG + Supabase Storage `product-images`。

## 実装済み vs 未実装・デモ

### 実装済み（MVP として動く）

- Next.js 16 App Router + Supabase Auth/DB/Storage
- Google ログイン、プロフィール、ロールガード
- 購入 4 ステップ / 消費 3 ステップ
- ホーム・店舗検索・詳細・マイボトル一覧・詳細
- admin / staff 価格申請フロー
- PWA 的メタ、モバイルファースト UI（M PLUS Rounded 1c）

### デモ・部分実装

- 特典・クーポン・ポイント（UI デモ、`benefit_news` のみ DB 連携）
- お気に入り店舗（sessionStorage）
- QR スキャン（見た目のみ）
- 消費トークン（クライアント生成・軽量対策レベル）
- ギフト（ログはあるが LINE 連携なし）

### 計画・README 記載だが未実装

- Stripe 本番決済
- LINE ログイン・ギフト送信
- 店舗確認用の時限署名トークン（本格的なもの）
- 割引率の店舗ページでの明示（戦略上重要だが UI 未完成の可能性）

### リポジトリの古い記述

- `README.md` の localStorage / `catalog.ts` — **実際は Supabase 移行済み**

## 不正対策（公式方針）

初期は **ガチガチにしない**。

- 使用済みフラグ / 残量管理
- 店舗側の目視（伝票と一致）
- 提示画面の軽いスクショ対策

## 技術スタック（把握している範囲）

- **Frontend:** Next.js 16, React 19, Tailwind v4, Framer Motion
- **Backend:** Supabase（Postgres, Auth, Storage）
- **Auth:** Google OAuth → `/auth/callback`
- **状態:** `StockProvider` + Server Actions（クラウド在庫）
- **マスタ:** `MasterDataProvider` + `/api/master-data`
- **デプロイ想定:** Vercel、`mybottle.app`

---

## デザイン・UX の方向性（会話から追加で分かっていること）

- **モバイルファースト**、ストレスのない余白
- **若者向け** — ポップ、キラキラ、グラデ、丸み（ただしトロピカルすぎ・地味すぎのバランス調整中）
- ログインは `start_page.png` 全面背景 + ロゴグロー + `bottle_pare.png`
- ホームのヒーローは `herocard.png` + 登録本数・残り杯数
- ボトルカードは `mb-bottle-stage--pop` のミント→クリーム台座で統一

## まだ分かっていない・要確認なこと

1. **実店舗でのパイロット結果** — 仮説検証ログはテンプレのみ（未記入）
2. **割引率の具体** — 店舗ごとに何 % か、UI でどう見せるか
3. **収益モデル** — 加盟店月額 / 取引手数料 / 広告等（コード・docs に未記載）
4. **PIN / 提示トークンの本番セキュリティ要件**
5. **Supabase RLS・migration の正本** — リポジトリ外管理の可能性
6. **LINE 連携の優先度と時期**
7. **YokaUnit と mybottle の組織関係・他プロダクトとの連携**

---

## 戦略サマリー（箇条書き）

1. **フック:** 割引 + ボトルキープの所有感  
2. **導入:** 店舗オペを変えない → 営業が早い  
3. **リテンション:** 残量・来店・特典でリピート  
4. **スコープ抑制:** 決済は店頭、アプリは証明と状態管理  
5. **地域密着:** 湘南から店舗データで実証  
6. **リスク回避:** DX しすぎない（最大の武器）
